# ADR 0003 — Vault update model: how notes evolve as the project grows

> A meta-architect design decision. Lean MADR form: it sets an internal documentation
> convention and crosses no security/data trust boundary, so the escalation triggers in
> `templates/adr.md` do not fire.

## Context and problem statement

A project is not a snapshot — it grows. Personas appear, the business reason sharpens, a
component's classification shifts. The vault has to absorb that change *predictably*, or it
rots into a record nobody trusts. The first draft over-rotated the other way: `PROJECT.md`
was marked `frozen`, conflating two different needs — "don't let the agent silently clobber
this on every re-run" (real) and "this can never change" (wrong). We need one clear rule for
how each kind of note handles updates, and a freshness/history trail so growth stays
traceable.

## Considered options

- **A — One rule for all notes** (e.g. "always update in place, never overwrite"). Simple,
  but wrong for *derived* notes like `INDEX.md` (which should be rebuilt from current state)
  and for *logs* like `CHANGELOG.md` (which must only ever append).
- **B — Three update modes, assigned per note type.** *(chosen)*

## Decision

Every vault note follows exactly one of three **update modes**:

1. **regenerated** — rebuilt from scratch each run; it is derived data, so overwriting is
   safe and correct. → `INDEX.md`, the generated `glossary.md` index.
2. **append-only** — new entries added, existing ones never edited. → `CHANGELOG.md`,
   `learning-log.md`, and ADRs (which evolve only by status transition, e.g.
   accepted → superseded, never by rewriting the original Decision).
3. **update-in-place** — edited surgically; the `updated` field is bumped and the change is
   logged to `CHANGELOG.md`; never blind-overwritten and never auto-regenerated by re-running
   a skill. → `PROJECT.md`, components, invariants, state-machines, `open-questions.md`,
   glossary-term notes.

History and freshness live in three complementary places, with no duplication: the `updated`
field (per-note freshness signal), `CHANGELOG.md` (the agent's own audit trail of writes),
and git (full diff history). Notes therefore do NOT carry inline revision logs — that would be
a second source of truth for history.

Consequently, `PROJECT.md` becomes `living` + update-in-place, and the `frozen` status value
is **dropped** from the canonical enum (YAGNI): it existed only to protect `PROJECT.md`, a need
now met by the update-in-place rule. It will be re-introduced — with a defined meaning and its
own ADR — the day we actually need to seal a record (e.g. preserving a `v1` snapshot beside an
evolving `v2`).

## Consequences

- **Good:** Growth is predictable — each note's update behaviour is knowable from its type.
  Derived notes stay correct, logs stay honest, hand-authored analysis is never clobbered.
  History is traceable without duplication.
- **Cost we accept:** The agent must classify each note into a mode and follow it; a wrong
  classification (e.g. treating an update-in-place note as regenerated) would lose content, so
  the mode mapping is stated once in the agent prompt and treated as authoritative.
- **Follow-ups:** Carry the three modes and the per-type mapping into the agent system prompt;
  each skill's idempotency rule must name the mode it uses for every file it writes.
