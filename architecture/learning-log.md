- 2026-06-11 · accepted deferral vs silent drift · [[2026-06-11-architecture-review]] · a deferral is legitimate while dated + re-pointable (ADR 0003 D5 → "consent gate in M2.S5"); it rots into silent drift the moment its landing milestone passes unmet and nobody re-dates it — a milestone-boundary review is what catches the slip before the note lies.
- 2026-06-11 · temporary invariant hand-off (INV-8→INV-1) · [[2026-06-11-architecture-review]] · a deliberately temporary invariant's value is making its removal a *tested* transition, not a quiet MERGE-creep; the guard enforcing "always CREATE" is swapped for "never merge without a human" and the test suite should witness the swap.
- 2026-06-11 · cheapest-first cascade (deterministic→embedding→LLM→human) · [[m3-cascade-matching]] · the §3.3 cascade orders checks by cost so tokens are spent only on genuinely ambiguous cases (Stage 3) and a human commits last (Stage 4) — the design force is "how little can we spend to be sure?", and the only graph-writing edges leave the human box (that visual *is* INV-1).
- 2026-06-11 · gate-the-write vs dedupe-after-write · [[m3-cascade-matching]] · the central M3 fork (DM6): intercept candidates before any graph write (only human-approved nodes ever exist) vs write-then-merge-duplicates (graph dirty until reviewed) — INV-1 ("only the author commits") pushes hard toward intercept-before-write, which retires INV-8 rather than layering on it.
- 2026-06-15 · vault-follows-source ordering · [[2026-06-15-architecture-review]] · a review may fold only drift with an *already-authoritative* basis (merged code/spec/PR); a decision resolved this same session whose authoritative home (PLAN_SHORT Decided) is written later must be *reported*, not pre-written into the vault — else the vault asserts what its own source of truth doesn't yet record, which is drift in the opposite direction.
- 2026-06-15 · intra-batch deduplication · [[intra-batch-dedup]] · the cascade only dedupes *cross-pass* (against the accepted graph at stage time), so a single first pass against an empty graph stages duplicates the queue can't merge — the gap a real browser walk surfaced; the cure (on-accept live re-match) is what makes §9 M3's "clean graph" true from the first run, not just eventually.
- 2026-06-15 · monotone refinement (suggestion-only automated writer) · [[m3s4c-intra-batch-rematch]] · re-match is the first automated process to mutate a *staged proposal* after staging — keeping it monotone (only new→merge, never the reverse) makes it idempotent and non-surprising, and the reason it doesn't threaten INV-9 is that INV-9 draws its line at *graph vs staging*, not *human vs machine*: a machine may freely rewrite suggestions, never the graph.
- 2026-06-16 · referential integrity · [[referential-integrity]] · an extracted relation names its endpoints as surface strings, but an entity id exists only after a human accepts — so the edge has nothing valid to reference at extraction time; writing it lazily (only once both endpoints resolve to *committed* ids) is what keeps the graph integral *and* makes the dreaded "re-point edges after a merge" mostly dissolve, because the edge is born pointing at the survivor rather than written-then-corrected.
- 2026-06-17 · state-machine symmetry · [[2026-06-17-architecture-review]] · when two features are duals (the node gate / the edge gate) their *models* should be too — S4e/S4f grew a real relation lifecycle (held / committable / written / rejected) in code while the vault drew only the entity twin; an asymmetric vault is what drift looks like before it bites, and the cure is to draw the missing machine, not to remember the gap (OQ-20).
- 2026-06-17 · provenance vs deduplication · [[2026-06-17-architecture-review]] · a content-derived id (uuid5 over the relation triple) deduplicates one-fact-to-one-edge but erases multiplicity (how many mentions, where) — the standard escape is to keep the dedup key on the edge and an occurrence table for the N:1 provenance, which is exactly what `staged_relations` already is, un-promoted to the graph; worth re-reading before any "show every passage where X betrays Y" feature.
- 2026-06-17 · span resolution (the null-offset gap) · [[m4-inline-highlights]] · inline highlighting looks like "render the offsets we stored" but `entity_mentions` carries null char offsets (the LLM path keeps an evidence quote, not offsets; the spaCy span that *has* them is discarded at accept) — so the first M4 slice is really a *where-does-this-entity-sit* problem, and the honest options trade a zero-backend render-time string search (inflection-blind, ambiguous on repeats) against persisting real spans (exact but a data change + backfill); naming the gap is what stops "just highlight the offsets" from being planned against data that doesn't exist.
- 2026-06-17 · read-only projection (stations mostly n/a) · [[m4-inline-highlights]] · a feature that only *reads* and renders mutates nothing, so most of the nine stations (Evidence, Expiry, Monitoring, Review) are legitimately n/a and it cannot violate the human-gate invariants (INV-1/9) — the architectural move is to *name* the empties as the signature of a read view rather than hunt for missing controls, and to carry only the one rule that does bite: highlight the committed world, never staged candidates (the read-side echo of INV-1).
- 2026-06-18 · derived state vs persisted state · [[relation-lifecycle]] · the edge gate's `held`/`committable` are *not* columns — only `staged|written|rejected` is persisted; held/committable are recomputed each read by re-resolving the surface endpoints against the current accepted set, so the same `staged` row shows two faces and can move between them freely as endpoints land or drift. Naming a derived state as a state is faithful to the *behaviour* (what the human sees, what the guard checks) even when no enum value exists for it — the twin candidate machine does the same with its in-memory cascade states; the discipline is to flag that the projection is recomputed, not written down, so nobody hunts for a `held` row in the table.
- 2026-06-18 · scan what you ship (gate scope = shipped risk) · [[software-composition-analysis]] · a security gate should reflect risk that actually *reaches a user*, so the frontend `npm audit` gate was scoped to `--omit=dev` — the SPA ships a static runtime bundle, so a `devDependency` (jsdom/vitest/eslint/Vite plugins) never crosses the [[trust-boundary]] to the shipped surface and auditing it gates `main` on code no user runs. Trigger: a HIGH `undici` advisory (GHSA-vmh5-mc38-953g) reached us only transitively via `jsdom` (the test DOM env), unreachable here, with no 14-day-soaked fix. The principled escape hatch for a genuinely-unreachable dep with no soaked fix is to **narrow the gate's scope to what ships** (an *adapt*, recorded in spec §6.7) — not waive the advisory and not force an unsoaked bump; this is [[defense-in-depth]] kept honest (dev-tool advisories stay visible to a bare `npm audit` + Dependabot, just un-gated). Authoritative home: spec §6.7 (amended 2026-06-18, M4.S1).
- 2026-06-18 · ego-graph (1-hop neighbourhood) · [[m4-side-panel]] · "a local graph *around* that entity" (§3.5) is the ego-graph — the focal node + its neighbours + the edges between them; the radius is a design call, and strict 1-hop (entity-incident edges only) is the legible choice for a narrow side panel because a 2-hop view of a busy entity is a hairball the slice can't yet filter. The build consequence is a *neighbourhood* query (edges incident to E + their far nodes), which Story Forge doesn't have yet — only whole-project `get_relations` — so the centre of gravity is again *where the data comes from* (a focused BFF endpoint vs composing the whole-graph fetch the viewer already does).
- 2026-06-18 · backend-for-frontend (BFF) · [[m4-side-panel]] · a purpose-built endpoint shaped to one screen's needs (the panel's details + occurrences + ego-graph bundle), rather than the client stitching several general-purpose endpoints — the same shape DM-IH-2 chose for the reader. The trade it buys: the cross-store join + neighbourhood filter live once on the server (Python-testable) and the client fetches only what a click needs, at the cost of a new endpoint; the alternative (compose client-side) is less code but loads the *whole* project graph into the reader to show one entity's neighbours. Deciding BFF-vs-compose is what sizes the slice (one slice vs a backend/frontend split).
- 2026-06-19 · lost update (write-side TOCTOU) · [[m4-entity-editing]] · the first graph-*editing* slice turns the side panel from a read view into a writer, and the new hazard is a lost update — two tabs (or a tab + a background refetch) each read then write, and the second silently clobbers the first with no error (the write-side cousin of [[toctou]]). The defences are last-write-wins (accept the clobber) or optimistic concurrency (a version/etag → 409); M4.S3a proposes LWW *named and accepted* at PoC because one local author makes true concurrency rare — the discipline is to name the anomaly as a conscious posture, not leave it an unconsidered blind spot.
- 2026-06-19 · first write flips the station fingerprint · [[m4-entity-editing]] · a read-only projection leaves Evidence/Review/Monitoring legitimately n/a and *cannot* break the human-gate invariants; the moment a slice *writes* committed state those stations go live, and the load-bearing one is Evidence — INV-3 says "every edit is reversible", which is impossible without a durable before→after record. So the architectural centre of a first-write slice isn't the UI affordances, it's (a) what new graph-writers exist and how the "only-N-writers" invariant (INV-9) is *reworded* not minted-against (the relation-write/ADR-0005 broadening precedent), and (b) where the undo before-image lives — the candidate/relation gates' `*_decisions` evidence rows are the pattern, but they're decision-keyed, so a graph edit needs its own record shape.
- 2026-06-20 · compound / transactional undo · [[2026-06-20-architecture-review]] · M4.S3a's INV-3 undo substrate is a *per-edit* before→after `graph_edits` row, which is exactly right for "entity E `type`: X→Y" or one edge add/remove — but the next slice (S3b) introduces the entity↔entity **merge**, which is *one* user action that is *many* writes (re-point every edge incident to B onto A — each a delete+recreate because the `uuid5(subject,predicate,object)` edge id changes — re-point B's `entity_mentions`, fold aliases/properties, delete B). Undoing it must reverse *all* of it together, so a flat per-row log no longer suffices: the before-image needs **grouping** (a merge id over its child rows, the compensating-transaction / saga shape) or undo replays children in reverse. The discipline the review surfaces: when a single command fans out to N writes, the *reversibility unit* is the command, not the write — design the before-image at the altitude of the action a human will say "undo" to.
- 2026-06-20 · compensating transaction (saga undo) · [[compensating-transaction]] · the M4.S3b decompose names the pattern behind that grouping: with no shared transaction across Neo4j + Postgres, "undo a merge/delete" isn't a DB rollback — you record each forward step's inverse and replay them in reverse under one `operation_id` ([[compensating-transaction]]). Two hazards earn names: an *incomplete before-image* is a non-reversible op masquerading as reversible (worse than no undo), and undo over *drifted state* is a [[lost-update]] in reverse (refuse, don't clobber). The payoff beyond undo: it resolves spec §10 q2 (graph versioning/rollback) the lightest honest way — the spec's own "append-only log of changes", *executed* — rather than full per-story snapshots.
- 2026-06-22 · materialization (derived → stored) · [[m4-s3c-manual-tagging]] · the reader's inline highlights are a *derived* projection — render-time string search over each accepted entity's name+aliases (DM-IH-1), with `entity_mentions` char-offsets left NULL. That bought real properties (a rename re-highlights for free; nothing to migrate; edit-robust) but at a cost the next slice exposes: a search hit is **anonymous** — no row, no id — so the author *cannot point at one specific highlight to correct it*, which is exactly what §3.5 manual tag/un-tag/change-boundaries demands. The architectural move S3c forces is **materialization**: turn (some of) the derived projection into stored, addressable records. The design lever is *how far* — materialize **everything** (a backfill migration; uniform corrections but you lose the derived view's self-healing) vs **incrementally** (store only the occurrences the author touches; keep search for the rest + a *suppression* record to subtract a rejected derived hit). The lesson generalises: a cheap derived projection and a correctable stored record are different shapes, and the boundary between them is a real decision — a half-materialized projection must be *reconciled* at read time, which is new complexity you're choosing to take on, not a free upgrade.
- 2026-06-23 · multi-tenancy / tenancy key · [[multi-tenancy]] · the narrowed multi-story slice forces apart two scopes that one-story-per-project let us blur — the **tenancy key** (`project_id`, *stored* on every Neo4j node, the hard isolation seam, spec §6.4) versus **per-story membership** ("which entities appear in this story", *derived* from the `entity_mentions → … → stories` FK chain, **not** stored — [[materialization]]). The payoff of keeping membership derived: a new story shares the tenant's graph (so known entities seed its extraction — the §B.4 incrementality test) *and* answers "what's in this story" as a read-time filter, with no second [[source-of-truth]] to sync. The teachable distinction the slice rests on: **seed scope (project — "what might recur") ≠ membership scope (story — "what actually occurred here")** — different scopes over the same nodes, which is why "add a story that reuses the graph" needs almost no new persistence.
- 2026-06-25 · doc-freshness as a state machine (4th occurrence) · [[2026-06-25-architecture-review]] · `overview.md` drifted in the *identical* way at four milestone boundaries (the 2026-06-17, -06-20, and now -06-25 sweeps each fixed the prior recurrence): an update-in-place note that nobody re-reads between rolls falls behind the as-built reality every time. The lesson is now strongly evidenced and structural, not "remember harder": a document has a lifecycle with transitions and guards exactly like the systems it describes, and "edited-but-not-re-stamped" / "milestone-shipped-but-overview-not-refreshed" is an *unguarded transition*. The forcing function that closes it is the milestone-roll sweep itself — four data points toward the still-open ADR-0002 question of wiring `review-architecture` into the roll ritual. The same forcing-function gap the vault already closed for review reports (resume step 3c) and security waivers (the `ignoreUntil` re-red), now seen for update-in-place strategic notes.
- 2026-06-26 · surrogate key vs natural (content-addressed) key · [[surrogate-key]] · Graph-quality S0 surfaces the §4 forward-compat call as an *identity* question: Story Forge edges use a natural key (`uuid5(subject,predicate,object)`), so the id *is* the fact — great for [[idempotency]]/dedup (ADR 0005) but the id *changes* the moment curation re-points an endpoint or consolidates a predicate, so nothing can be *attached to a specific edge across cleanup*. The architectural move §4 names is to reserve a **surrogate key** (an opaque `uuid4` carried alongside the natural key, preserved on re-point) — the natural key stays the MERGE/dedup key, the surrogate becomes the addressable handle a later qualifier hangs on. The discipline generalises: when an id doubles as both the dedup key *and* the thing you address, curation breaks the second role; splitting them (content key for dedup, surrogate for address) is cheap *at the moment every write path is already open* and turns a future migration into an additive change — but it forces a survivor rule (which handle wins when two edges fold).
- 2026-06-26 · reification (design the constraint, defer the feature) · [[reification]] · the §4 call is reification-shaped without building reification: turning an edge into a thing that can carry qualifiers (modality, temporal validity, n-ary arity, provenance — all deferred §5) needs the edge to have a stable identity to attach to, which a content-addressed edge can't keep across curation. So the only in-scope decision is whether to *reserve the hook* (the [[surrogate-key]] handle), not build any qualifier. The pattern this teaches — "design-constraint-now, feature-later" — is the owner's standing constraint made concrete: you do the cheapest thing that keeps the future additive (reserve an id) and explicitly build none of the future; the trap it avoids is baking in a bare, un-addressable triple a later model would have to unpick. A clean example of an architecture move that is *one property + one rule* now buying *no migration* later.
- 2026-07-01 · provenance (source-text trail, one-to-many) · [[provenance]] · Graph-quality S3 ("edge evidence") is the read side of the human gate — INV-1 is only as trustworthy as the context shown at the decision, so the slice surfaces each edge's [[provenance]]: the source paragraph(s) + evidence quote(s) that assert the fact. The teachable shape is that this trail is *distinct* from the decision-audit trail ([[compliance-audit-layer]], who-accepted-what for undo) — provenance points back to the *text*, audit points back to the *human action* — and that a content-addressed edge id makes provenance **one-to-many** (the same fact stated in N paragraphs is one edge but N source rows). The recurring architectural lesson (its 3rd Graph-quality instance after the side panel and navigation): the centre of gravity of a "show me the context" slice is *where the data comes from* — here the data exists (survives commit in `staged_relations`) but has no read path or index, so S3's real work is a focused BFF read, not new storage.
- 2026-07-06 · entity resolution (the umbrella; three moments, one signal) · [[entity-resolution]] · Graph-quality S4 makes explicit that Story Forge does entity resolution at *three* moments on one shared deterministic signal (RapidFuzz name + embedding cosine): at intake ([[cascade-matching]]), within a run ([[intra-batch-dedup]]), and now at curation time (S4 — the accepted graph deduped against *itself*). Naming the umbrella clarifies that S4 is not new machinery but the *same matcher turned inward* (each accepted entity scored against the others via a self-join over the `AcceptedSnapshot` the intake reader already assembles), and that the invariant uniting all three is "machine proposes, human commits" (INV-1) — S4 being the pass with no auto-merge branch at all.
- 2026-07-06 · connected components / cluster drift · [[connected-components]] · S4's central modelling call (DM-CD-1) is whether "duplicate cluster" means flat pairwise suggestions or transitive groups via union-find over the above-floor similarity edges. The teachable trap is *cluster drift*: connected-components chains transitively, so A≈B and B≈C pull A and C into one group even when A≉C (a common near-name bridges two unrelated identities). Since the merge is inherently pairwise (survivor + one absorbed), pairwise suggestions are the honest V1 unit and sidestep drift entirely; transitive grouping is a star-guarded refinement (match the representative, not merely some member), not a free win.
- 2026-07-06 · blocking (the O(n²) entity-resolution lever) · [[blocking]] · S4's self-join is O(n²) over accepted entities — trivial at PoC scale (~186 nodes) so S4 does not block, but the term is recorded (Layer 9) as the named revisit-lever: a large graph would partition into candidate blocks (a name prefix, a phonetic code, an LSH bucket over embeddings) and only score within-block pairs, trading a small recall risk for a large speed-up. The lesson is the discipline of *naming a deferred cost as a decision* rather than letting O(n²) become a silent surprise at scale.
- 2026-07-08 · direct manipulation drives the reuse call · [[direct-manipulation]] · Graph-quality S5 makes the graph canvas the editing surface — [[direct-manipulation]] (edit the object in place, not in a separate pane) applied to a knowledge graph. The teachable move is that a *UX principle selects the cheapest architecture*: because the author edits on the canvas, S5 should surface the write plumbing that already ships (the reader's `ReaderEntityPanel` + its BFF hooks) onto the canvas rather than fork a graph-native editor (DM-S5-1) — "edit in place" reads as "reuse the existing writers here," not "build new writers here." The whole slice is UX-surfacing except its one genuine gap (edge re-predicate/re-target), and that gap is genuine only because the content-addressed edge id (`uuid5(subject,predicate,object)`) *re-keys* on a change — which is exactly why S5b is the slice that finally consumes the reserved §4 [[surrogate-key]] handle: a client-side remove+add can't preserve a handle across the re-key, an atomic server op can. The lesson generalises: the "net-new" in an in-place-editing slice is never the editing UI, it's wherever the object's *identity* moves under the edit.
